## 1. 软件概述

FormuMind 接收一项研发需求（基材、耐盐雾目标、膜重、固化温度、清洗率、VOC 上限等），运行一个完整的研发闭环：

```
需求输入 → 专利/文献检索 → RAG 知识构建 → 推荐配方 → DOE 实验计划
        → 固化/界面模拟 → 贝叶斯闭环寻优 → Top-N 配方排行
                  ↑                                  │
                  └──────── DOE 实验结果回灌 ◄────────┘
                            （训练数据驱动的预测模型）
```

**核心价值**：提高配方成功率、缩短研发周期。随着真实实验数据的积累，平台会用数据驱动模型逐步取代经验代理模型，使推荐与寻优越来越贴近实测结果。

**三大产品域**：

| 产品域 | 中文 | 关键指标 |
|--------|------|----------|
| `anticorrosion_coating` | 防腐蚀涂料 | 耐盐雾时长、膜重、附着力、铅笔硬度 |
| `degreaser` | 脱脂剂 | 清洗率、泡沫指数、槽液寿命 |
| `surface_treatment` | 表面处理剂 | 膜重、耐盐雾、附着力促进指数 |

## 2. 设计理念：适配器 + 离线回退

平台的每一个外部重型引擎都置于一个**适配器**之后，并配有**确定性离线回退**实现。因此 FormuMind **今天即可端到端运行** —— 无需 GPU、无需 API 密钥、无需 C++ 工具链 —— 当真实引擎被安装后会自动"点亮"切换。

| 能力 | 真实引擎（可选） | 内置回退 |
|------|------------------|----------|
| 研究 / 对话 | 9 家大模型 —— Claude、OpenAI、Gemini、Grok、Meta(Groq)、DeepSeek、Qwen、Kimi、MiniMax | 基于知识库的规则合成 |
| 专利检索 | `patent_client` | 各产品域的精选种子语料 |
| 文献检索 | `arxiv`、`semanticscholar` | （离线时不返回额外结果） |
| 互联网检索 | `duckduckgo-search` | （离线时不返回额外结果） |
| 文件摄取 | `markitdown`（PDF/DOCX/XLSX/PPTX/HTML/图片…）→ `pypdf`/`python-docx` | 纯文本解码 |
| RAG 检索 | OpenNotebook 流水线 | 内存 TF-IDF 索引 |
| 接地问答 | ChemCrow 智能体（化学类问题）· paper-qa（语义合成） | TF-IDF 重排 → 配置的大模型 → 摘录 |
| 物性预测 | RDKit + DeepChem/ChemBERTa · MoLFormer（预留） | 透明的经验代理模型 |
| VOC / 密度 | `thermo` 质量加权密度 | 1.3 kg/L 名义假设 |
| 化合物数据 | PubChemPy（SMILES / 分子量） | 人工精选的原料库 |
| 寻优 | Summit (Bayesian/TSEMO) → Optuna (NSGA-II/TPE, CPU) | 纯 numpy UCB 贝叶斯寻优器 |
| 化学计量 | ChemFormula / RDKit | 自包含的分子式解析器 |
| 固化 / MD 模拟 | HTPolyNet · LUNAR · LAMMPS (Docker) | 解析近似 |

> 真正轻量且高价值的部分 —— **DOE 引擎**（全因子 / 部分因子 / Plackett-Burman / 中心复合 / 拉丁超立方）和**贝叶斯寻优器** —— 均以纯 numpy 真实实现。

## 3. 界面导览

界面为暗黑工业风、仿 NotebookLM 的三栏布局，将**输入 → 研究 → 输出**清晰分离：

```
┌──────────────────────────────────────────────────────────────────────┐
│  FormuMind · 金属表面处理配方研发平台      [⚙ 设置] [🕐 历史]         │
├──────────────────┬───────────────────────────┬───────────────────────┤
│  资料来源（左）   │  研究（中）                │  操作（右）           │
│                  │                            │                       │
│ · 研究主题       │  基于资料的接地问答：      │ 🧪 技术需求           │
│   （提示词框）   │  · 与已加载资料对话        │ ⭐ 推荐配方           │
│ · 来源类型：     │  · 每条回答附引用标签      │ 🔬 DOE 设计           │
│   ☑ 专利         │                            │ 📈 寻优收敛           │
│   ☑ 文献         │                            │                       │
│   ☑ 互联网       │  [向资料提问… ][发送]      │  （点击弹出对应窗口） │
│   ☑ 本地文件     │                            │                       │
│ [⬆ 上传][检索]   │                            │                       │
│ ── 已加载 (N) ── │                            │                       │
│ 📄 专利 · ✕      │                            │                       │
│ 📚 arxiv · ✕     │                            │                       │
│ 📎 本地 · ✕      │                            │                       │
└──────────────────┴───────────────────────────┴───────────────────────┘
```

- **左栏（资料来源）**：顶部是**研究主题提示词框**（为检索与问答提供上下文）；其下是来源类型复选框（专利 / 文献 / 互联网 / 本地文件）、文件上传按钮、**检索**按钮，以及已加载资料列表（每条可删除）。
- **中栏（研究）**：基于**已加载资料**进行问答的对话界面（TF-IDF 重排 → LLM 作答），每条回答下方以引用标签标注所用证据。
- **右栏（操作）**：四个按钮各自打开一个聚焦的**弹窗** —— 🧪 技术需求（产品域 / 基材 / 优化目标 / 约束）、⭐ 推荐配方（AI 推荐 Top-N）、🔬 DOE 设计（生成计划、填实测值、回灌训练、模型仪表）、📈 寻优收敛（运行贝叶斯闭环、收敛折线图）。按钮上的状态徽章显示运行中 / 结果条数。
- **标题栏**：⚙ **设置**（大模型供应商、模型、API Key、base URL）与 🕐 **历史**（会话快照抽屉，带实时条数徽章）。

## 4. 完整使用流程（闭环）

### 步骤 ⓪ 加载资料、围绕主题做研究（左栏 + 中栏）

1. 在**左栏**的提示词框输入研究主题（如"低 VOC 水性防腐蚀涂料"）。
2. 勾选要检索的来源类型 —— **专利**、**文献**（arXiv + Semantic Scholar）、**互联网**（DuckDuckGo），并/或**上传本地文件**（PDF/DOCX/XLSX/PPTX/HTML/图片，经 markitdown 解析）。点击**检索**。
3. 已加载资料显示在下方列表中。在**中栏**向这些资料提问 —— 回答基于已加载证据，并标注所引用来源。

> 离线时专利检索返回精选种子语料；文献与互联网检索需安装可选的 `intel` 依赖（见 §12）。

### 步骤 ① 设定需求并检索推荐（右栏弹窗）

四个研究功能以**右栏按钮**形式呈现，点击后各自弹出一个窗口，便于专注于单个任务。

1. 打开 **🧪 技术需求**：选择**产品域**（如"防腐蚀涂料"）与**基材**（如 `carbon_steel`），并设定优化目标、权重、方向与约束（如 VOC 上限）。
2. 打开 **⭐ 推荐配方**，点击**检索专利并推荐配方**。

平台将：
- 检索该域的专利/文献证据（离线种子语料）；
- 构建 TF-IDF RAG 索引并召回最相关条目；
- 基于知识库模板生成 3 个推荐配方变体（高活性 / 基线 / 精简），并按目标指标打分排序；
- 合成反应机理说明与研究对话。

**结果**：中栏显示机理 + 证据 + 对话；右栏排行榜显示 3 个推荐配方。展开任一配方卡可见其成分表、带不确定度的预测指标，以及 **3D 分子视图面板**（见 §5.9）。

![推荐配方 · 含分子视图](./images/05-recommend.png)

### 步骤 ② 运行寻优闭环

打开 **📈 寻优收敛** 弹窗并点击**运行寻优闭环**。平台异步运行贝叶斯多目标寻优（默认 24 次迭代）：

- 寻优器在关键配方杠杆（如锌系缓蚀剂含量、树脂、固化剂比例）的设计空间内采样；
- 每个候选配方经化学计量校验 + 经验/数据混合预测打分（多目标加权聚合）；
- 结果回灌优化器，迭代收敛。

**结果**：右栏上方显示收敛折线图；排行榜更新为 Top-5 寻优配方。

### 步骤 ③ 生成 DOE 并回灌实测结果

1. 在 **🔬 DOE 设计** 弹窗中选择设计类型（中心复合 CCD、Plackett-Burman 等），点击**生成 DOE**。
2. 系统生成实验记录表（每行一个实验，含各因子自然值 + 空白实测列）。
3. 两种回灌方式：
   - **手动**：直接在表格的"实测"列填入实验室测得的指标值，点击 **③ 回灌实验结果并训练模型**。
   - **批量**：点击**导出 CSV**下发给实验室 → 实验室填好 → 点击**导入 CSV**上传。
4. 当某指标累积样本数 ≥ `min_train_samples`（默认 4）时，平台自动训练该 (域, 指标) 的数据驱动模型。

**结果**：模型质量仪表盘显示 R² 半圆仪表 + RMSE；推荐与寻优自动切换为"经验 + 数据"混合预测。

> **闭环关键**：实测数据越多，数据驱动模型权重越高（混合权重 `w = n/(n+8)`），预测越贴近实测。

## 5. 核心功能详解

### 5.1 多目标寻优（加权帕累托聚合）

工业配方研发通常是多目标权衡 —— 既要高耐盐雾，又要低成本、低 VOC。每个产品域内置默认目标组合：

| 产品域 | 默认多目标 |
|--------|-----------|
| 防腐涂料 | 耐盐雾(最大化, 0.5) + 成本(最小化, 0.25) + 可持续性(最大化, 0.25) |
| 脱脂剂 | 清洗率(最大化, 0.5) + 成本(最小化, 0.3) + VOC(最小化, 0.2) |
| 表面处理 | 耐盐雾(最大化, 0.5) + 膜重(最大化, 0.2) + 成本(最小化, 0.3) |

每个目标经 min-max 归一化后按权重聚合，方向为"最小化"的目标自动取反，使得"得分越高越好"。可通过 API 的 `objectives` 字段自定义指标、权重与方向。

### 5.2 成本与可持续性评分

知识库中 26 种原料均带有 `price_cny_per_kg`（价格，元/kg）和 `voc_contrib`（挥发分比例 0–1）。每次预测都会计算：

- `cost_cny_per_kg` —— 按质量分数加权的配方成本；
- `voc_gpl` —— VOC 含量（g/L）；密度默认 ~1.3 kg/L，安装 `thermo` 后按真实质量加权混合密度计算；
- `sustainability_idx` —— 可持续性指数（0–100，对高 VOC 与高成本扣分）。

当配方 VOC 超过需求设定的 `voc_limit_gpl` 时，配方卡片会显示橙色警告。

### 5.3 预测置信区间（不确定性量化）

每个预测指标都附带一个 `predicted_std` 不确定度估计：

- **sklearn 随机森林**：用各决策树预测的标准差作为集成不确定度；
- **numpy ridge 回退**：用训练集 RMSE 作为保守的常数估计。

配方卡片中以 `值 ± std` 形式展示；当 std 超过值的 20% 时显示为橙色，提示该预测可信度较低。

### 5.4 收敛可视化

寻优运行返回 best-so-far 收敛曲线（`history`），右栏以 recharts 折线图渲染，X 轴为迭代次数、Y 轴为最优目标得分，悬停显示精确数值。

### 5.5 模型质量仪表盘

每个已训练模型显示一个半圆 SVG R² 仪表（>85% 绿色 / >60% 黄色 / 其余红色）+ 后端类型（sklearn-rf 或 numpy-ridge）+ 样本数 + RMSE + 交叉验证 R²。

### 5.6 多大模型供应商支持

研究合成与接地问答可运行在**九家大模型供应商**之上，从 ⚙ **设置**对话框（或环境变量）中选择。七家 OpenAI 兼容供应商共用一个 `openai` SDK，仅以各自的 `base_url` 区分；Claude 用 `anthropic`，Gemini 用 `google-genai`。

| 供应商 | 推荐模型 | SDK / base URL |
|--------|----------|----------------|
| Anthropic (Claude) | `claude-sonnet-4-6` | `anthropic` |
| OpenAI | `gpt-4o` | `openai` |
| Google (Gemini) | `gemini-2.0-flash` | `google-genai` |
| xAI (Grok) | `grok-2` | `openai` · `https://api.x.ai/v1` |
| Meta（经 Groq） | `llama-3.3-70b-versatile` | `openai` · `https://api.groq.com/openai/v1` |
| DeepSeek | `deepseek-chat` | `openai` · `https://api.deepseek.com` |
| Qwen（通义千问） | `qwen-plus` | `openai` · `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| Kimi（Moonshot） | `moonshot-v1-128k` | `openai` · `https://api.moonshot.cn/v1` |
| MiniMax | `abab6.5s-chat` | `openai` · `https://api.minimax.chat/v1` |

在设置中选择供应商与模型、粘贴 API Key（支持显示/隐藏）、按需填写自定义 base URL，点击**保存并测试连接**。Key 会在本次会话内同步到后端，并持久化在浏览器 localStorage。未配置 Key 时，全部功能仍可通过离线规则引擎运行。

### 5.7 多源检索、文件上传与接地问答

研究步骤被拆分为可勾选的多个来源，全部置于带离线回退的适配器之后：

- **专利** —— `patent_client`（离线：各产品域精选种子语料）；
- **文献** —— arXiv + Semantic Scholar；
- **互联网** —— DuckDuckGo（无需 API Key）；
- **本地文件** —— 上传 PDF/DOCX/XLSX/PPTX/HTML/图片；**markitdown** 将其转换为文本（回退到 `pypdf` / `python-docx`），再切分为证据片段。

所有勾选来源的结果会被合并、去重，并按相关度排序进入左栏资料列表。中栏对话随后基于这些资料**接地作答**：先用 TF-IDF 重排选出最相关证据，再由 LLM 组织答案，并在每条回复下以标签展示所用引用。

### 5.8 可选智能引擎（自动检测）

除大模型供应商外，FormuMind 在内置回退之上叠加了若干专业引擎。每个都**自动检测**：安装对应库即自动切换、无需改代码；未安装时继续走确定性离线路径，行为永不崩坏。

- **寻优分层** —— 闭环寻优器自动选择已安装的最强引擎：**Summit**（Bayesian/TSEMO，`heavy` extra）→ **Optuna**（NSGA-II/TPE，纯 CPU，轻量 `optimize` extra）→ 内置 numpy UCB 寻优器。所用 `engine` 会写入寻优结果。
- **接地问答路由** —— 化学类问题（LogP、溶解度、毒性、相容性、反应、结构…）在安装 **ChemCrow** 后路由至该智能体；其余问题走 **paper-qa** 语义合成（带页码引用）。两者均回退到 TF-IDF 重排 → 配置的大模型 → 摘录。
- **化合物富集** —— 设置 `FORMUMIND_ENRICH_COMPOUNDS=true` 且安装 **PubChemPy** 后，启动时自动补全原料库缺失的 SMILES / 分子量（人工精选值始终优先）。
- **VOC 物性接地** —— 安装 **thermo** 后，`voc_gpl` 由真实质量加权混合密度计算，替代 1.3 kg/L 名义假设。

### 5.9 3D 分子视图（预留）

每个展开的排行榜配方卡都带有 **3D 分子视图面板**。本版本为**占位符 + 数据契约**：列出配方中带 SMILES 的组分，并说明它们将通过 **3Dmol.js** 渲染为球棍模型。完整 WebGL 视图（以及用于增强物性预测的预留 **MoLFormer** 嵌入路径）将在后续升级中交付；占位符让当前打包保持轻量。

## 6. DOE 实验结果回灌与模型训练

这是平台的核心闭环机制。经验预测器只是一个先验，真实实验数据才是事实来源。

### DOE 设计类型

| 设计 | 用途 |
|------|------|
| `full_factorial` | 全因子，因子少时全面探索 |
| `fractional_factorial` | 部分因子，降低实验次数 |
| `plackett_burman` | 主效应筛选，最省实验 |
| `ccd` | 中心复合设计，建二次响应面 |
| `lhs` | 拉丁超立方，空间均匀填充 |

### 回灌与训练机制

1. 每条实验记录被特征化（`features.py`）为基于角色的组成 + 工艺向量。
2. 当某 (域, 指标) 累积样本 ≥ `FORMUMIND_MIN_TRAIN_SAMPLES`（默认 4），训练一个模型：
   - 装了 scikit-learn → `RandomForestRegressor`；
   - 否则 → 自包含 numpy ridge 回归。
3. `predictor.predict` 将训练模型与经验先验**混合**，样本越多模型权重越大（`w = n/(n+8)`）。
4. **数据是事实来源**：模型不做二进制持久化，启动时从持久化的实验数据集重建，保证可复现。

## 7. 数据导入导出

### DOE 实验记录表导出（B3）

- 在 DOE 区点击**导出 CSV** 或 **XLSX**，下载一份含空白 `measured_<指标>` 列的实验记录表，发给实验室填写。
- API：`GET /api/doe/{plan_id}/export?format=csv|xlsx`
- XLSX 需后端安装 `openpyxl`（可选 `[export]` extra）；CSV 用标准库，永远可用。

### 实验结果 CSV 导入（B3）

- 在 DOE 区点击**导入 CSV**，上传填好的记录表，批量入库并自动重训。
- API：`POST /api/experiments/import-csv`
- 解析规则：`measured_*` 列识别为实测值，`cure_temperature_c` 路由到工艺字段，空白行自动跳过，兼容 Excel 的 UTF-8 BOM。

### 配方导出（F2）

排行榜每张配方卡右上角的**导出 ▾**菜单提供：

- **复制 JSON** —— 复制完整配方对象到剪贴板；
- **导出 CSV** —— 成分行 + 预测指标导出为表格；
- **导出 PDF** —— 生成单页报告单（配方名、成分表、预测指标含不确定度、配方理由）。jsPDF 按需懒加载，不增加首屏体积。

## 8. 历史会话管理（F5）

- 每次完成**研究、寻优、回灌**后，平台自动保存一份会话快照（需求 + 排行榜 + 模型 + 收敛曲线 + 时间戳）。
- 点击标题栏**🕐 历史**按钮打开右侧抽屉，列出最近 20 条会话。
- 每条记录显示产品域标签、时间、Top-1 配方名与得分；点击即可**一键恢复**该会话状态。
- 支持**清空**历史。数据存储于浏览器 localStorage，刷新页面不丢失，无需后端。

## 11. 配置项

所有配置通过环境变量驱动（前缀 `FORMUMIND_`），均有安全的离线默认值。

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `FORMUMIND_LLM_PROVIDER` | `anthropic` | 当前供应商：`anthropic`/`openai`/`gemini`/`xai`/`groq`/`deepseek`/`qwen`/`moonshot`/`minimax` |
| `FORMUMIND_LLM_MODEL` | `claude-sonnet-4-6` | 当前供应商使用的模型 |
| `FORMUMIND_<供应商>_API_KEY` | 空 | API 密钥，如 `FORMUMIND_ANTHROPIC_API_KEY`、`FORMUMIND_DEEPSEEK_API_KEY`；为空时走离线规则合成 |
| `FORMUMIND_LLM_BASE_URL` | 供应商默认 | 覆盖 OpenAI 兼容 base URL |
| `FORMUMIND_SEARCH_LIMIT_PER_SOURCE` | `5` | 每个来源类型最多抓取的结果数 |
| `FORMUMIND_DB_URL` | `sqlite:///./data/formumind.db` | 实验数据库；可指向 Postgres |
| `FORMUMIND_REDIS_URL` | `redis://localhost:6379/0` | Celery broker |
| `FORMUMIND_CELERY_EAGER` | `true` | 无 broker 时在进程内同步执行任务 |
| `FORMUMIND_OPTIMIZE_ITERATIONS` | `24` | 寻优迭代次数 |
| `FORMUMIND_TOP_N_FORMULAS` | `5` | 排行榜配方数 |
| `FORMUMIND_MIN_TRAIN_SAMPLES` | `4` | 训练某指标模型的最小样本数 |
| `FORMUMIND_AUTO_RETRAIN` | `true` | 提交实验后自动重训 |
| `FORMUMIND_USE_CHEMCROW` | `true` | 安装后将化学类问题路由至 ChemCrow |
| `FORMUMIND_ENRICH_COMPOUNDS` | `false` | 启动时用 PubChem 补全 SMILES/分子量（需 `intel` + 网络） |

### 数据持久化（B5）

- 实验记录持久化于 SQL 数据库（默认 SQLite，零配置文件型）。
- 启动时若检测到旧的 `experiments.json`，会**自动迁移**进 SQLite（幂等，原文件保留作审计）。
- 多进程部署时把 `FORMUMIND_DB_URL` 指向 Postgres 即可，无需改代码。

## 12. 启用真实科学引擎

在有能力的机器上安装对应 extras，适配器会自动切换，无需改动代码：

```bash
pip install -e ".[llm]"          # Claude + OpenAI + Gemini SDK（覆盖全部 9 家供应商）
pip install -e ".[science]"      # scipy, scikit-learn, RDKit, ChemFormula, thermo
pip install -e ".[optimize]"     # optuna（纯 CPU 多目标寻优器，NSGA-II/TPE）
pip install -e ".[intel]"        # patent_client, paper-qa, chemcrow, pubchempy, arxiv, semanticscholar, duckduckgo-search
pip install -e ".[file_ingest]"  # markitdown, pypdf, python-docx（本地文件上传）
pip install -e ".[heavy]"        # torch, deepchem, transformers（MoLFormer）, summit, ase
pip install -e ".[export]"       # openpyxl（XLSX 导出；CSV 无需任何依赖）
```

安装 `science` extra 后：
- 物性预测会补充 RDKit 分子描述符特征；
- 模型训练自动从 numpy ridge 升级为 scikit-learn 随机森林（含集成不确定度）；
- 化学计量校验改用 ChemFormula 精确计算；
- `thermo` 让 VOC（g/L）按真实混合密度计算。

寻优器会自动选择已安装的最强引擎（**Summit** → **Optuna** → numpy），接地问答把化学类问题路由至 **ChemCrow**、其余走 **paper-qa**，两者均回退到 TF-IDF + 配置的大模型（见 §5.8）。以上全部可选 —— 每个引擎在安装后自动点亮。

## 13. 常见问题与边界说明

**Q：没有 API 密钥能用吗？**
能。所有功能在完全离线下端到端可用，仅 LLM 研究合成会使用基于知识库的规则引擎替代。

**Q：预测的性能数值可信吗？**
离线产出的是**工程合理的筛选估算值**，不是实验室验证的规格。请通过 DOE 回灌真实实验数据，让数据驱动模型逐步取代经验先验。

**Q：3D 仿真在哪里？**
每个排行榜配方卡现已带有 **3D 分子视图面板**，当前为占位符 + 数据契约（列出待用 3Dmol.js 渲染的带 SMILES 组分）。完整 WebGL 渲染 —— 以及通过 HTPolyNet/LAMMPS（Docker `heavy` profile）的反应性 MD 轨迹渲染 —— 将在后续升级中交付。寻优收敛图显示在「寻优收敛」弹窗中。

**Q：专利检索是真实在线抓取吗？**
默认使用各产品域的离线种子语料。真实 USPTO/EPO 抓取的适配器已就位，需配置 `patent_client` 等 `intel` extra 与相应密钥流程。

**Q：历史会话存在哪里？**
浏览器 localStorage（最多 20 条），不上传服务器。实验数据则持久化于后端 SQLite/Postgres。
